== 2. Persona Use Cases (Cockburn, User-Goal level)

=== 2.1 UC-1: Cache and retrieve a key-value pair

* *Primary Actor:* Application process (KV client).
* *Trigger:* the application calls `KVClient::Set(key, val, param)` to store a
  value and `Get`/`Del` to read or remove it [include/datasystem/kv_client.h:133,
  include/datasystem/kv_client.h:258, include/datasystem/kv_client.h:354].
* *Stakeholders & Interests:* the application wants low-latency, optionally durable
  storage; the operator wants bounded memory use.
* *Preconditions:* the client is connected to a local worker
  [src/datasystem/protos/share_memory.proto:170-176].
* *Main Success Scenario:*
  1. The value is written to shared memory (zero-copy)
     [include/datasystem/kv_client.h:133].
  2. Optionally a TTL is applied via `Expire`
     [include/datasystem/kv_client.h:441]; `Exist` checks presence
     [include/datasystem/kv_client.h:427].
  3. Batch variants `MSet`/`MGet` operate on multiple keys
     [include/datasystem/kv_client.h:216, include/datasystem/kv_client.h:312].
* *Extensions:* for `ExistenceOpt::NX`, an existing key returns `K_OK` and is left
  unchanged [include/datasystem/kv_client.h:130-133].
* *Postconditions:* durability follows the chosen WriteMode â€” cache-only, sync
  write-through, or async write-back
  [include/datasystem/object/object_enum.h:30-35,
  src/datasystem/common/l2cache/persistence_api.h:32-100].
* *Business Rules:* TTL of 0 means no expiration
  [include/datasystem/kv_client.h:52].

=== 2.2 UC-2: Publish and consume a reference-counted object

* *Primary Actor:* Application process (Object client).
* *Trigger:* the application increments the global reference and creates an object
  [include/datasystem/object_client.h:84,
  include/datasystem/object_client.h:130].
* *Preconditions:* the client is connected to a worker.
* *Main Success Scenario:*
  1. `GIncreaseRef` raises the cluster-wide reference count
     [include/datasystem/object_client.h:84].
  2. `Create` returns a shared-memory `Buffer`; `MemoryCopy` writes the payload
     [include/datasystem/object_client.h:130, include/datasystem/object/buffer.h:67].
  3. `Publish`/`Seal` makes the object immutable and visible
     [include/datasystem/object/buffer.h:79, include/datasystem/object/buffer.h:86].
  4. `Get` returns buffers for the object [include/datasystem/object_client.h:158].
  5. `GDecreaseRef` lowers the reference; the object is reclaimable at zero
     [include/datasystem/object_client.h:99].
* *Postconditions:* the object lives while its global reference count is greater
  than zero; `QueryGlobalRefNum` inspects it
  [include/datasystem/object_client.h:117].
* *Business Rules:* lifecycle is reference-counted across the cluster
  [include/datasystem/object_client.h:84, include/datasystem/object_client.h:99].

=== 2.3 UC-3: Produce and consume a stream

* *Primary Actor:* Application process (Stream client).
* *Trigger:* `CreateProducer` on a named stream
  [include/datasystem/stream_client.h:80].
* *Main Success Scenario:*
  1. The producer `Send`s elements [include/datasystem/stream/producer.h:59].
  2. A consumer `Subscribe`s [include/datasystem/stream_client.h:97].
  3. The consumer `Receive`s up to `expectNum` elements within a timeout
     [include/datasystem/stream/consumer.h:59].
  4. The consumer `Ack`s consumed elements
     [include/datasystem/stream/consumer.h:89].
* *Postconditions:* `DeleteStream` tears the stream down
  [include/datasystem/stream_client.h:117].

=== 2.4 UC-4: Move data between NPU device memory and host

* *Primary Actor:* Application process (Hetero client).
* *Trigger:* `MSetD2H`/`MGetH2D` to move hostâ†”device, `DevMSet`/`DevMGet` to move
  deviceâ†”device [include/datasystem/hetero_client.h:72,
  include/datasystem/hetero_client.h:83, include/datasystem/hetero_client.h:153,
  include/datasystem/hetero_client.h:165].
* *Main Success Scenario:* NPU HBM is abstracted as keyed objects; transfers use
  direct inter-card paths; `DevDelete` removes device objects
  [include/datasystem/hetero_client.h:182].

=== 2.5 Multi-interface user-goal journeys (proposed; team answer)

The four families above are the complete, correctly-leveled *system-interface*
catalog (team answer). The following cross-interface journeys are implied by the
public material and proposed for elevation to user-goal use cases (team answer);
the exact production journeys are *deferred* to the Domain Expert:

* *LLM inference KV-cache spill* â€” engine writes overflow KV cache â†’ multi-level
  HBMâ†’DDRâ†’SSD â†’ reload to raise hit rate / cut recompute, hetero H2D into HBM.
* *RL trainâ†”infer parameter rearrangement* â€” training writes parameters â†’
  fast cross-instance sync (object/hetero D2D/H2D) â†’ inference loads in seconds.
* *Checkpoint save/load* â€” training checkpoint â†’ durable WriteMode + L2 â†’ reload.
* *Stream data-flow* â€” producerâ†’consumer shared-memory paging replacing a broker.

== 3. System Use Cases (per technical interface)

=== 3.1 Clientâ†”worker object RPC (object POSIX service)

* *Input + validation:* `Create`/`Publish`/`Get`/`Delete` request messages
  [src/datasystem/protos/object_posix.proto:45,
  src/datasystem/protos/object_posix.proto:98].
* *Processing:* `WorkerOCServiceImpl` implements Put/Get/Delete, migration, and
  async operations
  [src/datasystem/worker/object_cache/worker_oc_service_impl.h:87-100].
* *Output + status codes:* results are reported through the status enum
  [include/datasystem/utils/status.h:28-109].
* *Error responses:* the per-RPC mapping of returnable codes and retry semantics
  is not declared in the proto contract; one documented contract is that a
  synchronous write-through `Set` hitting a request-timeout during distributed-disk
  failover takeover means "the slot is recovering, not that the write succeeded â€”
  retry under timeout rules or rely on upper-layer idempotency" (team answer);
  `K_BUFFER_DEPRECATED` [include/datasystem/utils/status.h:88] is a non-retryable
  "regenerate the data" condition (team answer). A complete per-RPC error matrix
  is *deferred* and must be authored from the handlers.

=== 4.1 Entity model

* *Object metadata* â€” `ObjectMetaPb` (object_key, data_size, version, life_state,

* *Stream element* â€” `Element`, with per-producer buffer pools at the worker

* *Topology / placement state* â€” a consistent hash ring of token slots, serialized
  via protobuf [src/datasystem/protos/hash_ring.proto,

=== 4.2 State and lifecycle rules

* *Persistence (WriteMode):* `NONE_L2_CACHE`, `WRITE_THROUGH_L2_CACHE` (sync),
  `WRITE_BACK_L2_CACHE` (async), plus evictable variants
  [include/datasystem/object/object_enum.h:30-35].
* *Consistency:* `PRAM` and `CAUSAL`
  [include/datasystem/object/object_enum.h:38-41].
* *Lifecycle:* TTL (KV), global reference counting (object), LRU eviction/spill at
  the worker [include/datasystem/kv_client.h:52,
  include/datasystem/object_client.h:84, include/datasystem/object_client.h:99,
  src/datasystem/worker/object_cache/worker_oc_eviction_manager.h:70].
* *Crash semantics:* in cache-only mode (`NONE_L2_CACHE`/`NONE_L2_CACHE_EVICT`)
  data on a crashed worker is lost by design and outstanding buffers are
  invalidated as `K_BUFFER_DEPRECATED`; reconstructible data should be
  recomputed/re-Put, while data that must survive requires a reliability WriteMode
  plus a configured L2 backend, noting write-back has a pre-flush loss window
  (team answer) [include/datasystem/utils/status.h:88,
  include/datasystem/object/object_enum.h:30-35].
