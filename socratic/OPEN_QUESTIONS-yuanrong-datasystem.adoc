=== Q1.2.2: Who is the intended operator role?
Category: business-context

The code ships dscli and K8s charts but does not say who runs the system
(platform/SRE team vs. self-serving application teams), nor what is in- or
out-of-scope for that operator.

*Your answer:*
PROPOSED (confirm with team). The operating model in the docs is a *platform /
infrastructure team* posture, not application-team self-service:

* Deployment is an infra task with privileged, cluster-wide prerequisites:
  multi-node SSH trust, a separately-deployed ETCD cluster, shared NFS / SFS
  Turbo / OBS for L2 reliability, huge-page configuration, and (for hetero) NPU
  driver mounts in the container — all operator concerns, not per-app

* The worker runs as a per-node DaemonSet (one data-plane process per node,
  k8s/helm_chart/datasystem/templates/worker_daemonset.yaml), which is the
  classic "platform team owns the node agent, app teams just link the SDK"
  division. Application teams consume it purely through the C++/Python/Java/Go
  SDK and `ConnectOptions(host, port)` — they do not run workers.
* In Huawei's own use (MetaERP, Xiaoyi AI platform), it is operated as shared
  Serverless infrastructure under a platform org, not self-served per team [S1][S2].

In-scope for the operator: cluster bring-up/scale, ETCD-or-Metastore choice, L2
backend + shared storage, security flags, metrics/Prometheus + dashboard
(default :9080), capacity/huge-page tuning. Out-of-scope (app-team owned):
WriteMode/consistency choice, key/object lifecycle, retry/idempotency.
DEFERRED: the *org-specific* split of duties (who owns ETCD, on-call boundary)
is site policy — Operations to confirm.

=== Q1.4: Why was this system built — what is the primary motivation?
Category: business-context

The code shows the mechanism; README_EN.md:49-53 lists scenarios (LLM KVCache,
M→N elasticity, RL parameter rearrangement, checkpoint save/load, microservice
state). Which scenario is the primary design driver, and what pain justified a
new system rather than an existing cache?

*Your answer:*
PROPOSED (well-supported by public sources). Motivation is the openYuanrong
"distributed kernel" thesis — make a cluster behave like one machine — and the
data system is the *near-compute multi-level cache* that gives that abstraction
its data performance: pool HBM/DRAM/SSD across the cluster and let same-node
access be zero-copy via shared memory, while cross-node access auto-syncs the
object to the local node so the next access is again local [S1][S2].

Primary design driver = *AI training/inference*, specifically two pains an
off-the-shelf cache (Redis/memcached) does not solve well:

1. *Redundant cross-node copies / Broker overhead.* Traditional message-broker
   and RPC paths incur multiple copies and high bandwidth use; the data system
   removes them with shared-memory pages + active push and zero-copy [S3].
2. *Heterogeneous (NPU/HBM) data movement.* Object semantics hide D2D/H2D/D2H
   Send/Recv and provide HBM-backed objects with a multi-level (HBM→DDR→SSD)
   cache to raise capacity and hit rate — no general cache does this [S2].

Concrete originating wins (Huawei Xiaoyi AI platform): distributed KV Cache
overall performance > 1x improvement; RL train→infer parameter sync cut from
minutes to seconds [S2]. LLM-inference KV-cache spill (overflow HBM KV cache into
the data system to raise hit rate and cut recompute) is the headline scenario
[S2][S6]. So: LLM KVCache is the lead driver; checkpoint/microservice-state are
secondary applications of the same mechanism.

=== Q4.9: Which quality characteristic has top priority?

=== Q2.6.4: What is the intended semantics of WriteMode=NONE on worker crash?
Category: business-context

status.h:88 (K_BUFFER_DEPRECATED) shows the code expects data loss in cache-only
mode on crash. Is that the contractually correct behavior for each scenario, and
what is the caller expected to do?

*Your answer:*
ANSWERED (docs confirm). NONE_L2_CACHE / NONE_L2_CACHE_EVICT are *cache-only and
provide NO L2 reliability* — this is intended, contractually-correct behavior,
not a bug (best_practices_for_reliabilty.md:26-29; object_enum.h:30-35). On a
worker crash the cache-only data on that worker is lost and outstanding
shared-memory buffers referencing it are invalidated, surfaced to the client as
`K_BUFFER_DEPRECATED` (status.h:88). Caller contract:

* If the data is *reconstructible / a pure cache* (LLM KV-cache spill,
  recomputable intermediate) → cache-only is the correct, fastest choice; on
  loss, recompute/re-Put. This is the expected default for inference KV cache.
* If the data must *survive crashes* (checkpoints, microservice durable state) →
  the caller MUST select a reliability WriteMode (WRITE_THROUGH /
  WRITE_BACK[_EVICT]) AND the operator MUST configure an L2 backend
  (`l2_cache_type` = distributed_disk | sfs | obs). Note WRITE_BACK has a
  failure window where not-yet-flushed data can still be lost
  (best_practices_for_reliabilty.md:23).

So "data loss on crash in NONE mode" is by-design; reliability is opt-in on both
the business side (WriteMode) and the cluster side (L2). DEFERRED only:
confirmation that each *named scenario's* chosen default matches this
(Domain Expert).

=== Q3.6.3: What is the intended steady-state read/write timing/back-pressure model?
Category: design-rationale

Hot-path components are visible (shm → worker → peer/L2) but the sequencing —
when a Get blocks, fans out to peers, or falls through to L2, and how
back-pressure is applied — is spread across handlers, not documented as one
runtime scenario.

*Your answer:*
PROPOSED (from code + public design). Steady-state path for a read:

1. *Local shared memory first.* If the object is resident on the local worker,
   the SDK reads it zero-copy from the mmap arena — no RPC, no copy [S2]
   (allocator.h; share_memory.proto). This is the fast path and does not block on
   the network.
2. *Cross-node fetch.* If not local, the worker resolves the primary location via
   the hash-ring metadata (cheap local map lookups, GetMasterAddr /
   GetPrimaryWorkerUuid) and pulls from the peer via `GetObjectRemote` /
   `BatchGetObjectRemote` (worker_object.proto:196-206). Per the public design,
   the fetched object is then *synced to the local node* so subsequent reads are
   again local [S2] — i.e. cross-node access is a pull+localize, not a per-read
   remote hit.
3. *L2 fall-through.* If configured and the object was spilled/persisted, it is
   served from the L2 backend via PersistenceApi (persistence_api.h;
   l2_cache_type distributed_disk/sfs/obs).

Back-pressure: there is *no hard admission control* on the read hot path; flow
is shaped by (a) fixed thread pools (RPC 16, OC 32, SC 128 — Q4.2), (b) rate
limits on *background* movement (L2 async write 200 MB/s, migration 40 MB/s —
Q3.10.2), (c) the 100 MB no-copy threshold switching large payloads to zero-copy
transfer, and (d) bounded async queues that "smooth overload rather than enforce
admission" (.repo_context logging design note). Writes follow WriteMode: cache
write to shm, then sync (WRITE_THROUGH) or async-queued (WRITE_BACK, rate-limited)
to L2. DEFERRED: a single authoritative sequence diagram with the exact
block/timeout points per handler should be drawn by the Architect; the above is
reconstructed, not a published spec.

=== Q3.9.MasterWorkerSplit.Rationale: Why a master/worker split over peer-to-peer?
Category: design-rationale

The split is in code but not the reasoning, nor the consistency/availability
trade-off it was meant to buy.

*Your answer:*
PROPOSED (partly inferred). The split buys a *single metadata/placement
authority* separate from the data plane:

* Workers own DRAM/SSD/HBM resources and the actual cached data and do the
  node-to-node push/pull; the coordination side (master, backed by ETCD or the
  Worker-0 Metastore) owns membership, the hash-ring topology, object-location
  metadata, heartbeat/health, and failover orchestration
  (cluster_management.md:5; QUESTION_TREE Q3.5.4/Q3.9). Public docs state cluster
  management "implements node discovery, health detection, failure recovery and
  online scaling" and "distributed metadata management" — a coordination role,
  not a peer mesh (cluster_management.md:5).

Why not pure P2P: a coordinated authority gives a consistent, single source of
truth for *where each object lives* and for *who has failed*, which makes
placement, online rebalancing, and failover decisions simple and globally
consistent — at the cost of a coordination dependency (ETCD or the Worker-0
Metastore, a known SPOF in Metastore mode — Q5.3). A P2P/gossip design would
trade that consistency for weaker, eventually-consistent placement and harder
failover reasoning. The chosen split therefore favors *consistency +
operational simplicity of recovery* over P2P availability. NOTE: the master is
metadata-only ("no data plane", Q3.5.4), so it is not on the data hot path —
the split does not cost read/write latency. DEFERRED: confirmation of the
original decision rationale is an Architect call; the consistency/availability
framing above is reconstructed.

=== Q3.9.EtcdVsMetastore.Rationale: When should ETCD vs. built-in Metastore be used?
Category: design-rationale

Both backends are implemented. When is each chosen, and is Metastore-on-Worker-0
a supported production posture or a single-node/test convenience?

*Your answer:*
ANSWERED (docs are explicit). cluster_management.md:13-51 states the choice:

* *ETCD* — when you already run ETCD, when multiple datasystem clusters share one
  metadata center, when you need cross-datacenter HA ("ETCD provides a more
  mature HA solution"), or when you want metadata decoupled from business
  components for independent ops. This is the *production-HA* posture.
* *Metastore* (built-in, ETCD-compatible gRPC, runs in Worker 0 as head node,
  others connect via `metastore_address`) — for *quick deployment,
  resource-constrained environments, and simplified ops* (fewer components)
  (cluster_management.md:34-51).

Is Metastore-on-Worker-0 production-supported? The docs position it as a
*convenience for resource-limited deployments*, and Worker 0 is a documented
coordination SPOF with *no documented automatic failover* — `metastore_head_node`
must be set manually (cluster_management.md:46,126; Q5.3). Reading: it is a
supported *mode*, but NOT an HA posture — for HA/production-critical metadata,
use ETCD. DEFERRED only: whether the team formally blesses Metastore for any
production tier is their policy call.
