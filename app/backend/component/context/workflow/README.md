# Context Export — gh-aw Workflow

This directory is the **canonical home of the context-export workflow**: the gh-aw workflow
source, the pre/post scripts, and the vendored parts driver.
`.github/workflows/context-export.{md,lock.yml}` is a **deployed copy** produced by
`compile.sh` — gh-aw's runtime stale-lock guard re-reads the `.md` next to the lock (a
missing/diverged source fails every run's activation job), and GitHub Actions only runs locks
from `.github/workflows/`. **Never edit the deployed `.md` or the lock by hand** — a CI drift
test (`app/tests/context-workflow-drift.test.js`) pins the canonical and deployed `.md`s
byte-identical and fails on any hand edit.

## Files

| File | Purpose |
|---|---|
| `context-export.md` | **Canonical** gh-aw workflow source (edit this one) |
| `compile.sh` | Deploy: copies the `.md` to `.github/workflows/` and runs `gh aw compile` |
| `scripts/locate.js` | Pre-agent step helper: locates the committed transcript in `.conversations/<owner>/<repo>/pr-<number>/` (scoped to THIS PR) |
| `scripts/assemble.js` | Post-agent step: assembles agent phase classifications → `SessionExport` JSON (`session-export.json`) |
| `scripts/parts-driver/` | Vendored bun driver: context-viewer claude parser + tiktoken (6-file cv/ subset + `{tiktoken,zod}` package.json); run in the pre-agent step under bun to produce the parts list |
| `.github/workflows/context-export.md` | Deployed copy (generated by `compile.sh`; do not hand-edit) |
| `.github/workflows/context-export.lock.yml` | Compiled lock (generated; do not hand-edit) |

## Deploy flow

After editing the canonical `.md` (or anything its compile depends on):

```bash
./app/backend/component/context/workflow/compile.sh
# On first compile or when adding new secrets, gh aw compile may need --approve — run it manually.
```

Commit **all three together**: the canonical `.md`, the deployed `.md`, and the lock.

## Required Actions secrets

| Secret | Purpose |
|---|---|
| `OPENAI_API_KEY` | OpenAI-compatible key used by the gh-aw agent (engine `codex`, model `gpt-5.5`; `CODEX_API_KEY` is an accepted alternative). Requests go to the private gateway in the lock's `OPENAI_BASE_URL`. |
| `GITHUB_TOKEN` | Auto-provided by Actions; used for `gh pr view` (read-only) |

Set `OPENAI_API_KEY` in **Settings → Secrets and variables → Actions → Repository secrets**.

## Token scopes for dispatch

To dispatch this workflow via the API (used by `core/actions.js`):

- **`actions:write`** — required to trigger `workflow_dispatch`
- **`actions:read`** — required to poll run status and download the artifact
- **`repo` (read)** — required to read PR data via `gh pr view`

## Safe-outputs mode

This workflow runs with `safe-outputs: { staged: true }`. In staged mode:

- The agent **cannot write** to the repository (no commits, no issues, no PR comments).
- All output is captured in `session-export.json` and uploaded as the `context-export` artifact.
- The noop tool is the only "write" the agent may call (it records a no-op event).

## Vendored parts driver

`scripts/parts-driver/` vendors a 6-file subset of context-viewer under `cv/` plus a
`package.json` declaring `{tiktoken, zod}` dependencies. The driver runs in the pre-agent
step under **bun** (bun's native TypeScript + ESM support; no compile step needed). It reads
the committed `.jsonl` transcript, invokes context-viewer's native `claude` parser, and emits
a JSON parts list consumed by the gh-aw agent.

The parts driver is a **spike / manual tool** — it is not part of the `npm test` offline suite
(which cannot run bun). The offline suite tests `locate.js` and `assemble.js` separately with
fakes. Only a live Actions run exercises the bun path end-to-end.

### Vendored context-viewer frontend (`app/frontend/context-viewer/`)

The prebuilt viewer `dist/` is vendored at `app/frontend/context-viewer/` (pinned commit
`abf784e`). The dashboard embeds it as a same-origin `<iframe>` that loads
`session-export.json` from the `/api/context/prs/:n/export.json` route. To rebuild:

```bash
# From a checkout of nilenso/context-viewer at abf784e:
bun install
NODE_ENV=production bun run vite build          # NODE_ENV=production sets base:/context-viewer/
# Then copy the CONTENTS of dist/ directly into app/frontend/context-viewer/ (not a dist/ subdir):
rm -rf app/frontend/context-viewer && cp -r dist app/frontend/context-viewer
rm -f app/frontend/context-viewer/404.html      # GitHub-Pages SPA shim; unused under Express
```

The vendored tree is ~8 MB (tiktoken WASM dominates) and is committed to the repo — it is not
rebuilt by `npm install` or `npm test`. `NODE_ENV=production` is **mandatory**: it is what sets
the Vite `base` to `/context-viewer/`; without it every asset 404s under the mount.

## Registration

GitHub only exposes `workflow_dispatch` for workflows present on the repository's **default
branch**; until the lock is there, the workflow won't appear under Actions → Run workflow and
an API dispatch returns 404. Set `CONTEXT_EXPORT_REF` on the app server to dispatch against a
specific branch whose tree contains both the lock and the `app/` scripts.

## Firewall-off advisory

Like preflight, this workflow runs the custody codex policy with a relaxed agent network
boundary (`network.allowed: [defaults, <gateway>]`) so the agent can reach the private
gateway. Under that policy **gh-aw can auto-file a spurious "failed" issue on a succeeded
run** — the app's run status (via SSE) and the downloaded `session-export.json` artifact are
the authoritative result. Ignore any auto-filed issue if the artifact is present and
well-formed.

See `docs/superpowers/specs/2026-06-07-sp4.6-agentics-discovery-findings.md` (option B) for
the full advisory.

## Manual acceptance (live run)

Dispatch normally happens **automatically** from the app — `pr.html` POSTs
`/api/context/prs/:n/trigger` when the pre-flight gate passes (no manual button). To validate
the workflow end-to-end without the app:

1. Ensure `OPENAI_API_KEY` is set in the repo's Actions secrets.
2. Ensure a `.conversations/<owner>/<repo>/pr-<number>/<session>.jsonl` transcript is committed on the PR branch.
3. Dispatch directly:

   ```bash
   gh workflow run context-export.lock.yml --ref <branch> -f pr_number=N -f repo=owner/repo
   ```

   (or **Actions → Context Export → Run workflow**).
4. Wait for the run to complete.
5. Download the `context-export` artifact and inspect `session-export.json`:
   - Confirm it is valid JSON with a top-level `files` array (≥1 entry, each with a
     `conversation.messages` list and a `colors` map) plus root `groups` and `analytics`.
   - Confirm `meta` echoes the judged `{pr_number, head_sha}` — the app fails loud on a
     `pr_number` mismatch.
   - The app's `shape.js` emits a `type:'phases'` record followed by 7 phase objects; check
     that at least one phase has `tokens > 0`.
6. Confirm **zero side-effects** — the PR must have NO new comments, NO new issues, NO commits
   added by the workflow run. Staged mode guarantees this; the acceptance run confirms it.
7. Load the app: confirm the PR row shows the per-phase token **bar + count**, the per-PR page's
   **"Conversation Token Composition"** panel shows the real composition, and clicking it opens
   **`context.html`** (the iframe lands on the conversation tab — click "Components" for the waffle).
8. Record the run URL and artifact link in the acceptance notes.
