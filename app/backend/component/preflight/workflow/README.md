# Preflight Gate — gh-aw Workflow

This directory is the **canonical home of all gate logic**: the gh-aw workflow source, the check
definitions, and the pre/post scripts the workflow invokes from the checkout.
`.github/workflows/preflight-gate.{md,lock.yml}` is a **deployed copy** produced by `compile.sh`
— gh-aw's runtime stale-lock guard re-reads the `.md` next to the lock (a missing/diverged source
fails every run's activation job), and GitHub Actions only runs locks from `.github/workflows/`.
**Never edit the deployed `.md` or the lock by hand** — a CI drift test
(`app/tests/workflow-drift.test.js`) pins the canonical and deployed `.md`s byte-identical and
fails on any hand edit.

## Files

| File | Purpose |
|---|---|
| `preflight-gate.md` | **Canonical** gh-aw workflow source (edit this one) |
| `compile.sh` | Deploy: copies the `.md` to `.github/workflows/` and runs `gh aw compile` |
| `checks.js` | Deterministic check implementations |
| `registry.js` | Check registry + `computeVerdict` (deterministic CLEAR/BLOCKED roll-up) |
| `scripts/deterministic-checks.js` | Pre-agent step: runs all LLM-free checks, writes `deterministic.json` + `ai-checks.json` |
| `scripts/locate.js` | Locates spec/plan artifacts in the PR diff |
| `scripts/merge-verdict.js` | Post-agent step: merges deterministic + AI results (+ the `meta` correlation echo from `pr.json`) into final `verdict.json` |
| `.github/workflows/preflight-gate.md` | Deployed copy (generated by `compile.sh`; do not hand-edit) |
| `.github/workflows/preflight-gate.lock.yml` | Compiled lock (generated; do not hand-edit) |

## Spec/plan association & severity

A spec/plan counts only when it is **associated with the PR** — committed in the PR's own diff,
or written into the PR description. `locate.js` does **not** attribute an unrelated doc found
elsewhere in the repo (doing so blocked unrelated PRs — e.g. a CI/registration PR — against
whichever doc sorted first under `docs/`). Consequently:

- **No associated spec/plan** → `spec-present` / `plan-present` emit a non-blocking **`warn`**
  (advisory), and `spec-adherence` / `plan-adherence` are **`skipped`** (adherence is not
  applicable without an artifact). The agent is not asked to judge adherence in this case.
- **An associated spec/plan exists** → `*-present` passes and the adherence checks run; if the
  code does not adhere, that adherence check **`fail`s** and the verdict is **BLOCKED**.

So a blocking `fail` on the spec/plan family means *"you declared a spec/plan and the code doesn't
match it"* — never *"this PR has no spec/plan"*. Use **Mark passed** to override per-PR when needed.

This logic is script-only (`locate.js` + `checks.js` + `deterministic-checks.js`), so changing it
needs **no lock recompile** — the workflow runs these from the PR-head checkout.

## Deploy flow

After editing the canonical `.md` (or anything its compile depends on):

```bash
./app/backend/component/preflight/workflow/compile.sh
# On first compile or when adding new secrets, gh aw compile may need --approve — run it manually.
```

Commit **all three together**: the canonical `.md`, the deployed `.md`, and the lock.

## Required Actions secrets

| Secret | Purpose |
|---|---|
| `OPENAI_API_KEY` | OpenAI-compatible key used by the gh-aw agent (engine `codex`, model `gpt-5.5`; `CODEX_API_KEY` is an accepted alternative). Requests go to the private gateway in the lock's `OPENAI_BASE_URL`. |
| `GITHUB_TOKEN` | Auto-provided by Actions; used for `gh pr view/diff` (read-only) |
| `GH_AW_GITHUB_TOKEN` | Optional PAT for cross-repo PR reads (if the PR is on a different repo) |

Set `OPENAI_API_KEY` in **Settings → Secrets and variables → Actions → Repository secrets**.

## Token scopes for dispatch

To dispatch this workflow via the API (used by `core/actions.js`):

- **`actions:write`** — required to trigger `workflow_dispatch`
- **`actions:read`** — required to poll run status and download the artifact
- **`repo` (read)** — required to read PR data via `gh pr view/diff`

## Safe-outputs mode

This workflow runs with `safe-outputs: { staged: true }`. In staged mode:

- The agent **cannot write** to the repository (no commits, no issues, no PR comments).
- All output is captured in `verdict.json` and uploaded as the `preflight-gate` artifact.
- The noop tool is the only "write" the agent may call (it records a no-op event).

## Registration

GitHub only exposes `workflow_dispatch` for workflows present on the repository's **default
branch**; until the lock is there, the workflow won't appear under Actions → Run workflow and an
API dispatch returns 404. Registration and execution are separate: once registered, a dispatch may
pass any `ref` whose tree contains both the lock and the `app/` scripts, and the run executes
**that** branch's tree (`core/actions.js` auto-resolves the default branch when no ref is given;
set `PREFLIGHT_GATE_REF` on the app server to dispatch a specific branch). If the workflow ever
de-registers, see the re-registration recipe in `app/README.md` ("Preflight via GitHub Actions").

## Manual acceptance (live run)

Dispatch normally happens through the app's **Trigger** button on the dashboard
(`POST /api/preflight/prs/:n/trigger`). To validate the workflow end-to-end without the app:

1. Ensure `OPENAI_API_KEY` is set in the repo's Actions secrets.
2. Dispatch directly:

   ```bash
   gh workflow run preflight-gate.lock.yml --ref <branch> -f pr_number=N
   ```

   (or **Actions → Preflight Gate → Run workflow**; `repo` defaults to `github.repository`).
3. Wait for the run to complete.
4. Download the `preflight-gate` artifact and inspect `verdict.json`:
   - Confirm `records[0]` is the checklist, every check has a `status`, and the `verdict` record's
     `status` is `clear` or `blocked` (not an error).
   - Confirm `meta` echoes the judged `{pr_number, head_sha}` — the app fails loud on a
     `pr_number` mismatch and adopts the judged `head_sha`.
   - If AI checks ran, confirm `ai-results.jsonl` entries are present in the verdict.
5. Confirm **zero side-effects** — the PR must have NO new comments, NO new issues, NO commits
   added by the workflow run. Staged mode guarantees this; the acceptance run confirms it.
6. Record the run URL and artifact link in the acceptance notes.
